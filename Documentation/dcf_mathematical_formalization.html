§0 - Provenance & Origin

From Guitars to General-Purpose Communication

Origin

The DeMoD Communications Framework began as a protocol for DeMoD Guitars - instruments with embedded computers designed and built by Asher, founder of DeMoD LLC. The original problem was deceptively specific: how do you move real-time audio, MIDI control data, and DSP parameters between an embedded processor inside a guitar body and external effects chains, amplifiers, and recording interfaces - with no handshake negotiation, no clock synchronization preamble, and no tolerance for latency?

That constraint - the receiver must extract meaning from a signal without prior coordination - turned out to be universal. The same mathematics that let a guitar's onboard processor identify an incoming effects-chain command without a USB enumeration handshake also lets an IoT sensor mesh, a game server network, or a distributed computing cluster identify and route packets without session negotiation. The protocol that emerged from DeMoD Guitars was formalized, validated by the United States Air Force as a secure communications framework, and generalized into the DeMoD Communications Framework (DCF) described in this document.

§1 - Motivation

What "Handshakeless" Means and Why It Matters

Most communication protocols begin with a negotiation phase. TCP performs a three-way handshake (SYN, SYN-ACK, ACK). TLS adds several more round-trips to agree on cipher suites and exchange certificates. USB devices enumerate. Bluetooth pairs. In every case, the sender and receiver must agree on timing, encoding, and identity before any useful data can flow.

DCF eliminates this phase entirely. A DCF receiver extracts the identity and content of a transmission from the signal itself, using the mathematical structure of the frame as the sole basis for detection. There is no SYN packet, no preamble sequence, no negotiated session state. The receiver simply listens, computes, and recognizes.

This is not magic - it is a direct consequence of a well-known result in harmonic analysis: the Fourier transform converts an unknown time delay into a phase rotation that can be discarded by taking the magnitude spectrum. The contribution of this document is to formalize that result in the specific context of DCF's 17-byte transport frame, to show how the 4-byte data payload is algebraically decoupled from the header during detection, and to prove that the framework is medium-agnostic by construction via category theory.

The intended audience is engineers, mathematicians, and protocol designers. Familiarity with linear algebra and basic signal processing is assumed; the category-theoretic sections (§5) are included for completeness and can be skipped without loss of practical understanding.

§2 - Wire Format

The 17-Byte Transport Frame and 4-Byte Payload

Before introducing the mathematics, it is essential to understand the concrete object being analyzed. Every DCF transmission begins with a fixed 17-byte header followed by a variable-length payload. The header is serialized in network byte order (big-endian) and contains four fields:

type 1 byte · u8
sequence 4 bytes · u32
timestamp 8 bytes · u64 (microseconds since epoch)
payload_len 4 bytes · u32
payload (data) N bytes (specified by payload_len · minimum 4 bytes for a command · maximum 1,454 bytes for UDP MTU safety)
0 1 5 13 17
FieldOffsetSizeRole in Spectral Analysis
type01 byte Message class identifier (0x01 HEARTBEAT, 0x02 TASK, 0x03 RESULT, etc.). Drawn from a small, fixed enum - contributes a deterministic spectral peak to the header's DFT. This is the primary detection feature.
sequence14 bytes Monotonically increasing counter. Contributes structured, slowly-varying spectral energy - the low-order bits cycle predictably, reinforcing header identification over multiple frames.
timestamp58 bytes Microsecond-precision epoch time. The high-order bytes change slowly (epoch scale); the low-order bytes exhibit quasi-periodic microstructure. Together they create a wideband spectral signature that is statistically distinct from random payload data.
payload_len134 bytes Declares the payload size. For fixed-size command payloads (e.g., 4 bytes), this field is constant - contributing another deterministic peak to the header spectrum.
payload17N bytes Application data. Content varies per message - contributes flat spectral noise floor under ensemble averaging. Invisible to frame detection; decoded independently after identification.

2.1   The 4-Byte Minimal Payload

The smallest meaningful DCF payload is 4 bytes (32 bits). This is the size of a single hardware plugin command (PluginRequest), a position coordinate component (f32), or a game event type with data byte. In the original DeMoD Guitar context, 4 bytes carried an effects-chain command: one byte for the target effect slot, one for the parameter index, and two for the 16-bit parameter value.

A frame with a 4-byte payload totals 21 bytes = 168 bits on the wire. This is the concrete object whose mathematical structure is formalized in the sections that follow.

2.2   Cross-Language Wire Compatibility

The same 17-byte header serialized by the C SDK via struct.pack(">B B I Q I", ...) is bit-identical to the output of the Rust ProtoMessage::serialize(), the Common Lisp HydraMesh encoder, and the Haskell SoapySDR pipeline. This is not an implementation accident - it is a mathematical consequence of the natural isomorphism proved in §5 below. The spectral fingerprint of a DCF frame is the same regardless of which language produced it.

§3 - The Signal Space

Unitary Fourier Operator on Locally Compact Abelian Groups

With the wire format established, we now construct the mathematical abstraction that makes handshakeless detection possible. The key insight is that a DCF frame is not merely a sequence of bytes - it is a signal, and signals live in Hilbert spaces where powerful structure-preserving transforms are available.

Specifically, we model the frame as an element of $L^2(\mathbb{G})$, the space of square-integrable functions over a locally compact abelian group $\mathbb{G}$. The choice of $\mathbb{G}$ depends on the physical medium:

  • For a continuous RF carrier (as in the DCF SDR pipeline): $\mathbb{G} = \mathbb{R}$.
  • For an acoustic channel (as in the original DeMoD Guitar application): $\mathbb{G} = \mathbb{R}$, with bandwidth constraints imposed by the instrument's transducer.
  • For digital packet transport over UDP, TCP, or gRPC: $\mathbb{G} = \mathbb{Z}/N\mathbb{Z}$, a finite cyclic group where $N$ is the frame length in bits.

The abstraction that unifies all of these is the unitary Fourier operator $\mathcal{F}$, defined over the Pontryagin duality between $\mathbb{G}$ and its dual group $\hat{\mathbb{G}}$:

Definition 3.1 - The Fourier Operator
$$\mathcal{F} : L^2(\mathbb{G}) \;\longrightarrow\; L^2(\hat{\mathbb{G}})$$
  • $\mathbb{G}$ is a locally compact abelian group - the domain in which the signal physically exists (time, space, or bit-position).
  • $\hat{\mathbb{G}}$ is the Pontryagin dual group - the space of continuous homomorphisms $\chi : \mathbb{G} \to \mathbb{T}$, where $\mathbb{T} = \{z \in \mathbb{C} : |z| = 1\}$ is the circle group. Informally, $\hat{\mathbb{G}}$ is the "frequency domain."
  • Unitarity means $\langle \mathcal{F}f,\, \mathcal{F}g \rangle_{L^2(\hat{\mathbb{G}})} = \langle f,\, g \rangle_{L^2(\mathbb{G})}$ - the inner product (and therefore all information content) is perfectly preserved. Nothing is lost in the transform; nothing is added. This is the mathematical guarantee that a handshakeless receiver can reconstruct the full frame without any side-channel information.

The explicit action of $\mathcal{F}$ on a signal $f \in L^2(\mathbb{G})$ is:

Definition 3.2 - Explicit Transform
$$\hat{f}(\chi) \;=\; \int_{\mathbb{G}} f(x)\, \overline{\chi(x)}\; d\mu(x), \qquad \chi \in \hat{\mathbb{G}}$$

The symbol $\mu$ denotes the Haar measure on $\mathbb{G}$ - the unique (up to scalar) translation-invariant measure. This is not a technicality. Translation-invariance of the measure is the formal reason why the Fourier transform's behavior does not depend on when or where the receiver begins sampling. The Haar measure is, in a precise sense, the mathematical embodiment of the word "handshakeless": it guarantees that $\hat{f}(\chi)$ is the same whether you start listening at time $t = 0$ or $t = \tau$ for any $\tau$.

In plain language

The Fourier transform takes a signal - a sequence of voltages on a wire, pressure waves in air, or bits in a buffer - and decomposes it into its constituent frequencies. The result is a "spectral fingerprint" of the signal. Two key properties make this useful for DCF: (1) the fingerprint is unique - different signals produce different spectra, so frame identification is unambiguous; (2) the fingerprint is computed using an integration measure that is indifferent to time-shifts, so the receiver does not need to know when the signal was sent.

§4 - Modulation Abstraction

Shift Invariance and Blind Synchronization

In any communication system, the signal arrives at the receiver with some unknown delay. In a handshake-based protocol, this delay is resolved during the negotiation phase - the parties agree on a time reference before data flows. In DCF, no such negotiation occurs. Instead, the receiver exploits an algebraic identity that converts the unknown delay into a form that can be either measured or discarded, depending on the application.

4.1   Translation Operator

The unknown arrival delay $a \in \mathbb{G}$ is modeled as the action of the translation operator, which shifts the signal in time without altering its content:

Definition 4.1 - Translation
$$\mathcal{T}_a f(x) \;=\; f(x - a), \qquad a \in \mathbb{G}$$

In the DeMoD Guitar context, $a$ is the propagation delay between the guitar's onboard processor and the external effects unit. In a game server network, $a$ is the one-way network latency. In an SDR pipeline, $a$ is the time-of-flight for the RF signal. In all cases, $a$ is unknown to the receiver at the moment of detection.

4.2   Modulation Operator

The corresponding effect in the frequency domain is the modulation operator, which encodes timing information as a phase shift without requiring a shared clock:

Definition 4.2 - Modulation
$$\mathcal{M}_\xi f(x) \;=\; e^{2\pi i \langle x,\, \xi \rangle}\, f(x), \qquad \xi \in \hat{\mathbb{G}}$$

4.3   The Commutation Relation

The fundamental identity of handshakeless communication is the commutation relation between the Fourier operator, the translation operator, and the modulation operator. It states that a time delay in $\mathbb{G}$ becomes a pure phase rotation in $\hat{\mathbb{G}}$:

Theorem 4.3 - DCF Commutation Relation
$$\boxed{\;\mathcal{F} \circ \mathcal{T}_a \;=\; \mathcal{M}_{-a} \circ \mathcal{F}\;}$$
  • $\mathcal{T}_a$ represents the unknown delay - the signal was sent at some time the receiver doesn't know.
  • $\mathcal{M}_{-a}$ represents the resulting phase rotation - a multiplicative factor of $e^{-2\pi i a \omega}$ at each frequency $\omega$.
  • This is an operator identity: it holds for every signal $f \in L^2(\mathbb{G})$, every delay $a \in \mathbb{G}$, and every group $\mathbb{G}$ in the LCA category. It is not an approximation.
In plain language

If someone sends you a message and you don't know exactly when they sent it, the only effect on the frequency-domain representation is that each frequency component gets "rotated" by an angle proportional to the delay. Critically, the strength of each frequency component is unchanged - only the angle changes. This means you can identify what was sent (by looking at the strengths) without knowing when it was sent (which only affects the angles).

This is the entire mathematical basis for handshakeless detection. Everything that follows is a consequence of this one identity.

4.4   Application to the 17-Byte Header

Consider a DCF frame arriving at the receiver with unknown delay $a$. The received signal is $\mathcal{T}_a f$, where $f$ is the transmitted frame. By the commutation relation, the Fourier transform of the received signal is:

Received Spectrum
$$\mathcal{F}\{\mathcal{T}_a f\}(\omega) \;=\; e^{-2\pi i a \omega}\; \hat{f}(\omega)$$

The magnitude spectrum is therefore:

Magnitude Invariance
$$\left|\mathcal{F}\{\mathcal{T}_a f\}(\omega)\right|^2 \;=\; \left|e^{-2\pi i a \omega}\right|^2 \cdot \left|\hat{f}(\omega)\right|^2 \;=\; 1 \cdot \left|\hat{f}(\omega)\right|^2 \;=\; \left|\hat{f}(\omega)\right|^2$$

The factor $|e^{-2\pi i a \omega}|^2 = 1$ for all $a$ and $\omega$, because complex exponentials live on the unit circle. The magnitude spectrum of the received frame is identical to the magnitude spectrum of the transmitted frame, regardless of the delay. The receiver computes $|\hat{f}(\omega)|^2$ from the incoming bitstream and compares it against the known spectral templates for each DCF message type - and the match succeeds regardless of when the frame was sent.

§5 - Category-Theoretic Structure

The Fourier Transform as a Natural Isomorphism

This section is addressed to readers with background in category theory and may be skipped without loss of practical understanding. Its purpose is to prove that DCF's handshakeless property is not merely a feature of a particular group ($\mathbb{R}$ or $\mathbb{Z}/N\mathbb{Z}$) but is an inherent structural property of the Fourier transform itself - guaranteeing that DCF works identically across every medium and encoding.

5.1   The Functors

We define two covariant functors from the category $\mathbf{LCA}$ of locally compact abelian groups (with continuous homomorphisms) to the category $\mathbf{Hilb}$ of Hilbert spaces (with bounded linear maps):

Definition 5.1 - Signal and Spectral Functors
$$\text{Sig} : \mathbf{LCA} \to \mathbf{Hilb}, \qquad \mathbb{G} \mapsto L^2(\mathbb{G})$$ $$\text{Spec} : \mathbf{LCA} \to \mathbf{Hilb}, \qquad \mathbb{G} \mapsto L^2(\hat{\mathbb{G}})$$

$\text{Sig}$ assigns to each group the Hilbert space of signals over that group. $\text{Spec}$ assigns to each group the Hilbert space of spectra over its dual. Both functors are well-defined because the Pontryagin duality functor $(-)^\wedge : \mathbf{LCA} \to \mathbf{LCA}^{op}$ is exact.

5.2   The Natural Isomorphism

The Fourier transform provides a natural isomorphism between these functors:

Theorem 5.2 - Naturality of the Fourier Transform
$$\boxed{\;\Phi : \text{Sig} \;\xRightarrow{\;\sim\;}\; \text{Spec}\;}$$

Concretely, for every continuous homomorphism $\varphi : \mathbb{G}_1 \to \mathbb{G}_2$, the following diagram commutes:

Naturality Square
$$\begin{array}{ccc} L^2(\mathbb{G}_1) & \xrightarrow{\;\mathcal{F}_{\mathbb{G}_1}\;} & L^2(\hat{\mathbb{G}}_1) \\[6pt] \Big\downarrow\scriptstyle{\varphi^*} & & \Big\downarrow\scriptstyle{\hat{\varphi}^*} \\[6pt] L^2(\mathbb{G}_2) & \xrightarrow{\;\mathcal{F}_{\mathbb{G}_2}\;} & L^2(\hat{\mathbb{G}}_2) \end{array}$$

The maps $\varphi^*$ and $\hat{\varphi}^*$ are the pullbacks induced by $\varphi$ and its Pontryagin dual $\hat{\varphi}$ respectively. Commutativity means: $\mathcal{F}_{\mathbb{G}_2} \circ \varphi^* = \hat{\varphi}^* \circ \mathcal{F}_{\mathbb{G}_1}$.

Why this matters for DCF

This naturality is the formal proof that DCF is medium-agnostic by construction. The homomorphism $\varphi$ might be:

  • An analog-to-digital converter mapping $\mathbb{R} \to \mathbb{Z}/N\mathbb{Z}$ (RF carrier to digital samples).
  • A resampling operation mapping $\mathbb{Z}/N\mathbb{Z} \to \mathbb{Z}/M\mathbb{Z}$ (changing the sample rate).
  • The embedding of a Minecraft redstone comparator's signal range $\mathbb{Z}/15\mathbb{Z}$ into the general nibble group $\mathbb{Z}/16\mathbb{Z}$.

In every case, the naturality square guarantees that the spectral identity of the DCF frame is preserved across the mapping. A frame detected over RF has the same spectral fingerprint as the same frame detected over UDP, over acoustic, or over redstone. This is not an engineering convenience - it is a theorem.

§6 - The Handshakeless Identity

Spectral Density as the Phase-Free Invariant

The commutation relation (§4.3) tells us that an unknown delay produces a phase rotation in the spectral domain. The receiver eliminates this unknown phase by computing the power spectral density - a quantity that depends only on the magnitude of the spectrum, discarding all phase information:

Definition 6.1 - The Handshakeless Identity (Wiener-Khinchin)
$$\boxed{\;\mathcal{S}_{xx}(\omega) \;=\; \lim_{T \to \infty}\; \frac{1}{2T}\; \mathbb{E}\!\left[\,\left|\, \mathcal{F}\{x_T(t)\}\,\right|^2\,\right]\;}$$
  • $x_T(t) = x(t) \cdot \mathbf{1}_{[-T,T]}(t)$ is the signal windowed over the observation interval $[-T, T]$. As $T \to \infty$, the window captures arbitrarily many frames.
  • $\mathbb{E}[\cdot]$ denotes the expectation (statistical average) over the ensemble of received signals, accounting for noise, multipath, and other channel impairments.
  • $|\cdot|^2$ takes the squared magnitude, collapsing all phase information: $|e^{-2\pi i a \omega} \hat{f}(\omega)|^2 = |\hat{f}(\omega)|^2$ for any delay $a$. This is the step that eliminates the need for a handshake.
  • $\mathcal{S}_{xx}(\omega)$ is a function of frequency alone - no phase, no clock reference, no handshake.

Equivalently, via the Wiener-Khinchin theorem, $\mathcal{S}_{xx}$ is the Fourier transform of the autocorrelation function $R_{xx}(\tau) = \mathbb{E}[x(t)\overline{x(t+\tau)}]$:

Wiener-Khinchin Duality
$$\mathcal{S}_{xx}(\omega) \;=\; \mathcal{F}\{R_{xx}\}(\omega) \;=\; \int_{-\infty}^{\infty} R_{xx}(\tau)\, e^{-2\pi i \omega \tau}\, d\tau$$
In plain language

The power spectral density answers the question: "how much energy is present at each frequency?" It deliberately throws away the timing information (the phase) and keeps only the structural information (the energy distribution). For DCF, this is the ideal representation: it preserves everything the receiver needs to identify the frame type while discarding the one thing the receiver doesn't know - when the frame was transmitted.

The Wiener-Khinchin duality adds a second perspective: $\mathcal{S}_{xx}$ can also be obtained by asking "how similar is the signal to a time-shifted copy of itself?" (the autocorrelation), and then Fourier-transforming the answer. This is the mathematical basis for DCF's RTT-based peer grouping: peers within the <50ms RTT threshold exhibit strong autocorrelation at lags corresponding to the frame period, which is how Dijkstra's RTT-weighted grouping algorithm clusters nodes without a handshake.

§7 - Spectral Separation of Header and Payload

Why the 4-Byte Payload Is Invisible to Frame Detection

The 17-byte header and the 4-byte payload play fundamentally different roles in the spectral analysis of a DCF frame. Understanding this separation is essential to understanding how DCF achieves both handshakeless detection and reliable data transport in the same frame.

7.1   Header: Structured Spectral Peaks

The header fields exhibit strong statistical regularity across multiple frames. The type byte is a constant drawn from a small enum. The sequence number increments monotonically, meaning its low-order bits cycle with a predictable period. The timestamp field has microsecond-epoch structure - its high-order bytes change slowly (on the scale of seconds), while its low-order bytes exhibit quasi-periodic microstructure. The payload_len field is often constant for a given message class (e.g., always 4 for hardware plugin commands).

When the spectral density $\mathcal{S}_{xx}$ is computed over an ensemble of received frames, these regularities produce persistent spectral peaks - frequency components that are consistently present from frame to frame. These peaks form the DCF header's spectral fingerprint.

7.2   Payload: Flat Noise Floor

The payload bytes, by contrast, carry application data that varies freely from message to message. An I2C command, a position coordinate, a game event - the content is determined by the application, not by the protocol. From the receiver's perspective (before decoding), the payload is effectively random.

Under the expectation operator $\mathbb{E}[\cdot]$, random data contributes a flat spectral noise floor - energy distributed uniformly across all frequencies with no persistent peaks. This is a standard result from the theory of wide-sense stationary processes: if the payload bytes are independent and identically distributed, their power spectral density is constant.

7.3   Detection by Threshold

The receiver performs frame detection by computing the magnitude spectrum of the incoming bitstream and looking for spectral peaks that rise above the noise floor. Because the header contributes persistent peaks and the payload contributes only noise, the detection process depends entirely on the header. The payload content is irrelevant to detection - it is invisible in the spectral domain.

This separation is what allows DCF to carry arbitrary data payloads without compromising the handshakeless property. The receiver identifies the frame by its header spectrum, then reads the payload from the known bit positions within the detected frame. Detection and decoding are two independent operations.

7.4   Algebraic Decomposition: The Tensor Product

The spectral separation of header and payload has a precise algebraic formulation. A complete DCF frame with a 4-byte payload occupies $17 + 4 = 21$ bytes $= 168$ bits. The bit-level group is $\mathbb{Z}/168\mathbb{Z}$, but the header and payload occupy disjoint subranges, so the frame is more naturally modeled as the direct product:

Frame Group Decomposition
$$\mathbb{G}_{\text{frame}} \;=\; \underbrace{\mathbb{Z}/136\mathbb{Z}}_{\text{header (17 bytes)}} \;\times\; \underbrace{\mathbb{Z}/32\mathbb{Z}}_{\text{payload (4 bytes)}}$$

The Discrete Fourier Transform over a direct product of finite groups decomposes as a tensor product of the individual transforms:

Theorem 7.4 - DFT Tensor Product Decomposition
$$\boxed{\;\mathcal{F}_{\mathbb{Z}/136\mathbb{Z} \,\times\, \mathbb{Z}/32\mathbb{Z}} \;\cong\; \mathcal{F}_{\mathbb{Z}/136\mathbb{Z}} \;\otimes\; \mathcal{F}_{\mathbb{Z}/32\mathbb{Z}}\;}$$
  • The left-hand side is a single 168-point DFT over the full frame.
  • The right-hand side decomposes it into a 136-point DFT over the header tensored with a 32-point DFT over the payload.
  • The tensor product $\otimes$ means these two transforms are algebraically independent: the spectral coefficients of the header do not depend on the payload content, and vice versa.

This decomposition has direct engineering consequences:

  • The receiver can run a 136-point DFT on just the header bits to perform frame detection, without computing the payload's spectrum at all.
  • Once the frame is detected, the receiver runs a separate 32-point DFT on the payload bits to extract data content.
  • These two operations can execute in parallel - and in DCF's plugin architecture, they do: the transport layer validates the header while the hardware or application plugin independently processes the payload.

7.5   Further Factorization via the Chinese Remainder Theorem

The number 168 factors as $168 = 8 \times 3 \times 7$. By the Chinese Remainder Theorem for finite abelian groups:

CRT Factorization
$$\mathbb{Z}/168\mathbb{Z} \;\cong\; \mathbb{Z}/8\mathbb{Z} \;\times\; \mathbb{Z}/3\mathbb{Z} \;\times\; \mathbb{Z}/7\mathbb{Z}$$

This factorization enables a multi-dimensional DFT decomposition that can be exploited for computational efficiency (reducing a single 168-point DFT to three smaller transforms of sizes 8, 3, and 7) and for structural analysis (each prime factor corresponds to a distinct spectral "axis" that can be examined independently). The Cooley-Tukey FFT algorithm is a direct application of this decomposition.

§8 - Discrete Specialization

The DCF Frame over Finite Groups

For DCF's digital transport layer, we specialize from the general LCA setting to the finite cyclic group $\mathbb{G} = \mathbb{Z}/N\mathbb{Z}$. The Pontryagin dual of $\mathbb{Z}/N\mathbb{Z}$ is itself: $\widehat{\mathbb{Z}/N\mathbb{Z}} \cong \mathbb{Z}/N\mathbb{Z}$, so the DFT is an automorphism on the same finite group.

Definition 8.1 - Discrete Fourier Operator on the DCF Frame
$$\mathcal{F}_N[k] \;=\; \sum_{n=0}^{N-1} x[n]\; e^{-2\pi i \, nk / N}, \qquad k \in \mathbb{Z}/N\mathbb{Z}$$

For $N = 136$ (17-byte header alone) or $N = 168$ (header + 4-byte payload), this is the standard DFT. The commutation relation discretizes to:

Theorem 8.2 - Discrete Commutation
$$\mathcal{F}_N\!\left[\mathcal{T}_m\{x[n]\}\right] \;=\; e^{-2\pi i \, mk/N}\; \mathcal{F}_N\!\left[x[n]\right]$$

And the discrete handshakeless identity - the spectral fingerprint used for frame detection - becomes:

Definition 8.3 - Discrete Spectral Signature
$$\mathcal{S}_{xx}[k] \;=\; \frac{1}{N}\;\left|\,\mathcal{F}_N[k]\,\right|^2 \;=\; \frac{1}{N}\;\left|\sum_{n=0}^{N-1} x[n]\, e^{-2\pi i nk/N}\right|^2$$
Mapping to DCF v5.0 transport

The DCF header maps directly onto the finite group $\mathbb{Z}/136\mathbb{Z}$. Each field occupies a sub-range of bit positions, and the spectral signature $\mathcal{S}_{xx}[k]$ acts as a structural fingerprint: the receiver identifies a DCF frame by its spectral shape, regardless of when (time-shift) or where (phase offset) it arrives.

The 4-byte payload maps onto $\mathbb{Z}/32\mathbb{Z}$. After the header's spectral fingerprint is matched and the frame is identified, the receiver extracts the payload from bit positions 136-167 and processes it through the appropriate plugin - a hardware command, a position update, a game event, or an effects-chain parameter, depending on the type field identified during header detection.

§9 - Complete Expression

The DCF Handshakeless Communication Theorem

Assembling §3-§8 into a single statement:

The DCF Theorem

Let $\mathbb{G}$ be a locally compact abelian group, $\hat{\mathbb{G}}$ its Pontryagin dual, $\mathcal{F}: L^2(\mathbb{G}) \to L^2(\hat{\mathbb{G}})$ the unitary Fourier operator, $\mathcal{T}_a$ translation by $a \in \mathbb{G}$, and $\mathcal{M}_\xi$ modulation by $\xi \in \hat{\mathbb{G}}$. Then:

$$\underbrace{\mathcal{F} \circ \mathcal{T}_a = \mathcal{M}_{-a} \circ \mathcal{F}}_{\text{Commutation (blind sync)}} \quad\Longrightarrow\quad \underbrace{\left|\mathcal{F}\{f\}\right|^2 = \left|\mathcal{F}\{\mathcal{T}_a f\}\right|^2}_{\text{Shift-invariant identity}} \quad\Longrightarrow\quad \underbrace{\mathcal{S}_{xx}(\omega) = \lim_{T\to\infty} \frac{1}{2T}\,\mathbb{E}\!\left[|\mathcal{F}\{x_T\}|^2\right]}_{\text{Handshakeless detection}}$$

Moreover, $\mathcal{F}$ is natural with respect to $\mathbf{LCA}$:

$$\Phi : \text{Sig} \;\xRightarrow{\;\sim\;}\; \text{Spec}$$

guaranteeing that this identity holds across all group representations - continuous ($\mathbb{R}$), discrete ($\mathbb{Z}/N\mathbb{Z}$), and $p$-adic ($\mathbb{Z}_p$) - making DCF medium-agnostic by construction.

For a frame with header $\mathbb{H} = \mathbb{Z}/136\mathbb{Z}$ and payload $\mathbb{P} = \mathbb{Z}/32\mathbb{Z}$:

$$\mathcal{F}_{\mathbb{H} \times \mathbb{P}} \;\cong\; \mathcal{F}_{\mathbb{H}} \otimes \mathcal{F}_{\mathbb{P}}$$

Header detection and payload decoding are algebraically independent operations.
